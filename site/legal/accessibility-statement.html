Accessibility Statement

LoyaltyCards v1.0.2+8  ·  Last Updated: July 20, 2026  ·  Compliance Target: WCAG 2.1 Level AA

The known dark-mode text-legibility risk has been checked and ruled out (see "Dark Mode" below, verified July 20, 2026). Broader VoiceOver/semantic-labeling work remains open — see Roadmap.

LoyaltyCards is committed to ensuring digital accessibility for all users, including those with disabilities. We strive to continually improve the user experience for everyone and apply relevant accessibility standards.

Current Status

🟡 Partial Compliance. LoyaltyCards is partially conformant with WCAG 2.1 Level AA — some parts of the application do not yet fully conform.

Implemented

  • Standard Flutter/Material widgets, inheriting iOS system accessibility features (AssistiveTouch, Voice Control, Switch Control, Zoom, Bold Text, Reduce Motion)
  • Minimum 16px body text, clear heading hierarchy, high-contrast text on backgrounds
  • Primary text meets/exceeds WCAG AAA contrast; buttons meet at least 4.5:1
  • Icons paired with text labels; visible button/focus states; clear navigation structure
  • Minimum 44×44pt touch targets, adequate spacing, large QR tap areas
  • Standard iOS gestures only — no complex or gesture-only actions
  • Face ID / Touch ID with passcode fallback always available; VoiceOver announces authentication prompts

Partial Implementation

Screen Reader Support (VoiceOver)

Basic support via iOS defaults — button labels, navigation changes, and text fields are read. Gaps: custom widgets may lack semantic labels, QR scanner status isn't clearly announced, stamp counts may be read as separate numbers, and business logos may be described as "image" rather than by business name.

Dynamic Type

Body text scales with system settings; some headings use fixed sizes; QR code size doesn't scale (functional limitation); card layout may break at extreme text sizes (200%+).

Dark Mode

The app follows the device's system light/dark appearance (both apps' main.dart wire up real, distinct light/dark ColorScheme objects, defaulting to system mode), but there is no in-app toggle to choose Light/Dark/System independent of the device setting. Verified July 20, 2026: every text-color usage most likely to cause an illegible-text bug in dark mode (21 instances across both apps) was checked against its background — all of them pair fixed text with a fixed background (branded info/success/warning badges), so none go illegible in dark mode. This is a deliberate, pre-existing pattern, not an oversight. The visible effect is that these badges keep a fixed light background regardless of theme, so in dark mode they can look like a light-colored card on an otherwise dark screen — a style inconsistency, not a legibility problem.

Not Yet Implemented

  • Comprehensive semantic labels/grouping on custom widgets; better QR scanner state announcements
  • Automatic announcements for success/error states and loading states (live regions)
  • Color customization, high-contrast override, or option to disable brand colors
  • Zoom/scaling of fixed card layouts; simplified UI mode
  • Audio cues or haptic alternatives to visual indicators (standard iOS haptics on button presses are supported)

Known Accessibility Barriers

QR Code Scanning (visual requirement)

Users with visual impairments cannot scan QR codes independently, and there's currently no non-visual alternative. Workaround: request assistance from business staff, or have the business scan the customer's QR code and verbally confirm details. Voice-guided scanning and manual code entry are potential future enhancements, not currently scheduled. NFC as a non-visual alternative would require architecture changes and is not on the current product roadmap.

Card Visual Design

Cards use color and layout to convey information, which can be harder for color-blind or screen-reader users. Business name and stamp count are always present as text, mitigating this partially.

Limited Screen Reader Optimization

VoiceOver support relies on iOS defaults rather than custom optimization; navigation flow may not be fully optimal for screen reader users. All buttons have text labels and navigation is linear and predictable as a baseline mitigation.

Fixed-Color Badges Don't Adapt to Dark Mode

See "Dark Mode" above — branded callout badges keep a fixed light background regardless of theme (verified as a style issue, not a legibility one), and users cannot force a light/dark mode independent of their device setting.

Testing Performed

Manual testing has covered basic VoiceOver navigation, Dynamic Type at system sizes, Reduce Motion, manual WCAG contrast checks, and touch target measurement. No automated accessibility testing or CI/CD checks are in place yet, and no formal audit or testing with users who have disabilities has been performed yet.

Roadmap

Before the final pre-submission build

  • Verify the dark-mode text-legibility risk — done July 20, 2026, confirmed not present
  • Add Semantics widgets to all interactive elements; optimize VoiceOver announcements and card-detail labels
  • Improve QR scanner accessibility (voice guidance); test with real VoiceOver users

Medium priority (may defer post-v1.0)

  • Give fixed-color badge containers theme-derived variants for dark-mode visual consistency (polish, not a legibility issue)
  • Decide on an in-app Light/Dark/System override vs. following the system setting only

Under consideration (not committed, post-v1.0)

  • Voice-guided card selection; simplified UI mode; customizable high-contrast themes
  • Professional third-party accessibility audit; WCAG 2.1 Level AA certification

Legal Compliance Context

Relevant standards we track: Americans with Disabilities Act (ADA) Title III and Section 508 in the US; WCAG 2.1 and EN 301 549 internationally; Apple's iOS Human Interface Guidelines and App Store Review Guideline 2.5.18 (Accessibility). We are actively working toward full WCAG 2.1 Level AA compliance.
